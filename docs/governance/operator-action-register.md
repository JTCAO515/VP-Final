# Operator Action Register

Status: active

This register is the durable source of truth for actions that require the accountable operator in an
external console, legal account, secret manager, or production environment. Repository code uses
named placeholders and must remain honestly unavailable until the listed verification passes.

When the operator asks for setup help or the associated objective reaches its release gate, produce a
beginner-friendly Chinese tutorial for every unresolved row. Never ask the operator to commit or paste
a secret into GitHub, documentation, an Issue, a PR, or application logs.

## Register Schema

Every action MUST have a stable ID, explicit status (`open`, `in-progress`, `verified`, `blocked`, or
`retired`), associated Issue, purpose/capability, accountable owner, environment, prerequisites,
placeholder names, exact external action, unblock/verification evidence, rollback, and last-reviewed
date. Update the register in the same PR that creates, changes, verifies, retires, or makes an action's
assumptions false.

## Sanitized External Evidence — 2026-07-24

This observation supersedes only the dated QStash/Upstash absence statement above. Codex inspected
Vercel environment-variable names and deployment behavior without reading or retaining any value,
cookie, raw IP address, prompt, or provider payload.

- `vp-final-web` Preview and Production now declare the four OA-011 QStash setting names and all
  OA-012/OA-013 Upstash, limit, and IP-hash setting names. Safe metadata records those names as
  updated on 2026-07-23. `vp-final-ops` correctly does not declare these Web-runtime-only settings.
- A fixed, non-PII request with an intentionally invalid `upstash-signature` reached the production
  completion callback and returned HTTP 401 with `Invalid delivery signature.` This proves the
  deployed callback fails closed before database processing. It does not prove a genuine signed
  delivery, replay rejection, or durable completion.
- Fourteen parallel production Copilot requests from one real network address used distinct forged
  `x-forwarded-for` values. Eight passed the IP guard and then reached the known database-authentication
  failure; six returned HTTP 429 with `Retry-After: 1`. This proves the deployed trusted-address
  limiter is shared and atomic enough for the observed window, and that forged `x-forwarded-for`
  values did not create independent buckets. It does not prove the stored Redis key/TTL shape or the
  absence of a model trace/cost row after rejection.
- OA-012's three-successful-turns/fourth-turn-wall acceptance remains blocked by OA-004: requests
  cannot complete a real turn while the production database rejects authentication. OA-011's genuine
  signed delivery and OA-013's Redis key/TTL and trace/cost checks also remain outstanding.
- After the operator reported replacing `DATABASE_URL`, Codex rebuilt both Web and Ops Production.
  The new Web Production request and a separate authenticated Preview probe still exposed only the
  allowlisted PostgreSQL code `28P01`. The probe logged no SQL parameter, connection string, user
  content, or credential and was removed immediately after observation. The credential replacement
  therefore remains unverified and OA-004 is still the controlling blocker.
- A read-only Supabase Management API query through the authenticated CLI confirmed that
  `public.pois` and `supabase_migrations.schema_migrations` exist. Remote migration history contains
  14 versions and stops at `20260713182322`; the repository contains 25 migrations, so the 11 reviewed
  migrations from `20260715015825` through `20260721033000` are not applied remotely. No application
  row or credential was read. This proves a separate schema lag but does not supersede the runtime
  `28P01` credential blocker.
- A second temporary Preview probe parsed `DATABASE_URL` only into booleans. It proved the value is
  configured, URL-parseable, Postgres-shaped, and uses the approved project username, but it is not a
  port-6543 Transaction Pooler URI and its password component still contains the literal
  `YOUR-PASSWORD` placeholder. No URL component or credential was logged. The probe was removed and
  not committed. This is the exact cause of the observed `28P01`.

## Closure Rule

An action closes only when its external evidence, date, environment, verifier, and linked Issue/PR are
recorded here. A configured placeholder without verification remains open. Secret values are never
recorded; evidence names the setting and observed behavior only.

Use the [operator tutorial template](operator-action-tutorial-template.md) when a tutorial trigger is
reached. This register remains authoritative for status; a generated tutorial is an execution aid.
